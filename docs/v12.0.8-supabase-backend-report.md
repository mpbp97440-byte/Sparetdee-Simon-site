# MPBP440 V12.0.8 — backend Supabase, phase 1

## Périmètre

Ce lot ajoute uniquement l'infrastructure backend. Aucune page HTML, URL, ancre, vidéo, lecteur, animation, image, logo ou feuille de style n'est modifiée. Les deux fichiers locaux protégés hors Git sont conservés.

## Architecture

- `supabase/config.toml` initialise le projet Supabase local sans référence de projet ni secret dans le dépôt.
- `supabase/migrations/20260719000100_mpbp440_analytics.sql` est la source de vérité versionnée du schéma, des RLS, des index et des RPC.
- `assets/js/mpbp440-supabase.js` est le service unique prêt à être chargé par une future couche d'intégration non visuelle. Il ne contient aucune clé et reste inerte sans configuration publique de runtime.
- `.github/workflows/supabase-migrations.yml` valide chaque migration sur une pile Supabase isolée. Sur `main` uniquement, il lie le projet avec les secrets GitHub et exécute `supabase db push`.

Les secrets `SUPABASE_ACCESS_TOKEN`, `SUPABASE_PROJECT_REF` et `SUPABASE_DB_PASSWORD` ne sont lus que comme variables d'environnement du job de migration. Ils ne sont ni affichés, ni écrits, ni requis localement.

## Tables et protection

| Table | Usage | Protection |
| --- | --- | --- |
| `sessions` | Identifiant anonyme de session | RLS, accès RPC uniquement |
| `site_visits` | Une visite par session | RLS, accès RPC uniquement |
| `video_views` | Une vue par clip et session | RLS, accès RPC uniquement |
| `video_likes` | Un like actif par clip et session | RLS, accès RPC uniquement |
| `artist_pages` | Une visite par artiste et session | RLS, accès RPC uniquement |
| `event_views` | Une visite par événement et session | RLS, accès RPC uniquement |
| `analytics_daily` | Agrégat journalier public | RLS, lecture anonyme seulement |

Les contraintes d'unicité assurent l'anti-refresh. Les quatre identifiants de clips et les trois artistes existants sont explicitement autorisés. Les écritures directes anonymes sont interdites : les fonctions `SECURITY DEFINER` constituent l'unique voie d'écriture atomique.

## RPC et analytics

- `increment_site_visit()` : une seule visite réelle par session.
- `increment_video_view()` : point de comptage par clip, à appeler après 10 secondes de lecture ou à la fin.
- `toggle_video_like()` : ajoute ou retire le like unique de la session.
- `increment_artist_visit()` et `increment_event_visit()` : comptage dédupliqué.
- `get_video_engagement()` : total de vues et likes par clip.
- `get_analytics_overview()` : visites, pages vues, clips, likes, top artistes et top clips sur 24 h et 30 jours.

Le service JavaScript fournit retry borné, file d'attente hors ligne (maximum 50 événements), reprise au retour du réseau, cache de session et gestion d'erreur sans incidence sur l'interface. Il n'est intentionnellement pas injecté dans les pages de ce lot : cette contrainte conserve le rendu et le comportement actuel strictement identiques tant qu'une configuration publique de runtime et une intégration produit n'ont pas été explicitement validées.

## Tests et statut

- Syntaxe JavaScript : `node --check assets/js/mpbp440-supabase.js`.
- Cohérence des espaces : `git diff --check`.
- Validation de migration en CI : `supabase start` puis `supabase db lint --level warning`.
- Migration du projet lié : `supabase db push --linked --password ...`, autorisée uniquement après validation CI sur `main`.

Le poste local ne possède pas le CLI Supabase. La version officielle est installée dans GitHub Actions afin que les tests et la liaison ne chargent jamais les secrets localement.
